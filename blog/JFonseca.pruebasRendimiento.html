Pruebas de estrés desde la perspectiva de los sistemas lineales e invariantes en el tiempo¶

Juan M. Fonseca-Sols Agosto 2020 5 min read

Resumen¶

Las pruebas de rendimiento de software son tiles para determinar si un sistema es capaz de procesar suficientes cantidades de informacin sin caerse o realentizarse. Ejemplos de sistemas que requieren hacer este tipo de pruebas son las pginas web de los bancos o de los sitios de comercio electrnico, donde un retraso en el despliegue de la informacin puede traducirse en prdida de clientes. En este ipython notebook estudiamos un tipo de prueba de rendimiento llamada "pruebas de estrs" que tiene como objetivo encontrar el punto de quiebre, es decir, el mximo nmero de transacciones por segundo que el sistema es capaz de procesar antes de caerse. A diferencia del resto de artculos, nuestro enfoque se dedica a la parte del modelado matemtico y sistemas lineales e invariantes en el tiempo (LTI, por sus siglas en Ingls), en lugar de explicar las herramientas de software existentes para este tipo de pruebas. Se identificaron dos modelos para simular el procesamiento del sistema ante una entrada en forma de rampa, pero, se determin que los servidores web no cumplen con la definicin de un sistema lineal e invariante en el tiempo.

1 Pruebas de estrés¶

Una prueba de estrs tiene como objetivo encontrar el punto de quiebre, es decir, el maximo nmero de transacciones por segundo que el sistema es capaz de procesar antes de caerse [1]. Para encontrar este valor se utiliza un enfoque iterativo que incrementa linealmente en el nmero del nmero de transacciones por segundo para monitorear el nmero de errores por segundo y los tiempos de respuesta. El nmero de transacciones por segundo tambin es llamado usuarios virtuales por segundo (o VU/s, por sus siglas en ingls) y grficamente se ve as:

Figura 1. Imgen tomada sin permiso de https://abstracta.us/wp-content/uploads/2015/10/why-perf-testing-is-nec-chart-768x463.png.

Esta forma escalonada de la funcin de usuarios virtuales por segundo puede representarse como una rampa descrita por la ecuacin:

$$ x[n]= \frac{dx[n]}{dn} n u[n],\qquad(1) $$

donde $0<\frac{dx[n]}{dn}$ es la tasa de cambio (VU por segundo) y $u[n]$ es la funcin de Heaviside o escaln unitario (1 cuando $0 \le n$, 0 en el resto). Tomando $N=100$ como la duracin del experiento podemos hacer:

El nmero de usuarios virtuales por segundo se relaciona con otra curva llamada ancho de banda del sistema (throughput, en Ingls), que representa la capacidad del servidor a travs del tiempo. Esta curva es linealmente creciente en los primeros instantes, pues entre ms se le exige al sistema ms capacidad mostrar, y luego se comporta de forma constante, cuando llega al punto de quiebre. Para modelarla, podemos tomar la ecuacin (1) y hacer $\frac{dx[n]}{dn}=\frac{q}{n_0}$, como sigue:

$$ x[n] = \frac{q}{n_0} n u[n], $$

y luego definir la respuesta del sistema a la rampa:

$$ y[n] = \min(x[n], q),\qquad(2) $$

donde $0<q$ es el punto de quiebre y $n_0$ el instante donde ocurre. Tomando $q=5$ y $n_0=60$ podemos ver el siguiente grfico:

N = 150 # duracion del experimento (en numero de muestras)
n0 = 60 # instante del punto de quiebre (en numero de muestras)
q = 5.0 # punto de quiebre (en usuarios virtuales por segundo)1
n = np.arange(0,N)
x = q/n0 * n

def respuestaRampa(x):
    N = len(x)
    y = np.zeros(N)
    for n in range(1,N):
        y[n] = np.min([x[n], q])
    return y

y = respuestaRampa(x)

Despus de haber transcurrido $n_0$ muestras, la respuesta del sistema se aplana como es esperado y la cantidad de errores aumenta proporcionalmente al nmero de transacciones no procesadas. Este nmero de errores por segundo es tambin otra medida de inters y se calcula como $e[n] = x[n] - y[n]$, la diferencia entre el nmero de usuarios por segundo y el ancho de banda del sistema. Los errores en un servidor web corresponden a solicitudes con cdido de respuesta 500, como lo define el protocolo de transferencia de hipertexto (HTTP, por sus siglas en Ingls).

De la cantidad de errores por segundo y la cantidad de usuarios virtuales por segundo es posible definir otra medida llamada tasa de error, calculada como:

$$ E[n]=e[n]/x[n]. $$

Esta medida permite identificar cuando el sistema se cuentra estresado, porque si $\frac{dE[n]}{dn} \propto \frac{dx[n]}{dn}$ significa que la cantidad de errores incrementa proporcionalmente a la cantidad de usuarios virtuales por segundo y el sistema est ignorando las solicitudes de los usuarios que superan el punto de quiebre, como se observa a continuacin despus del instante $n_0$:

Hay otra medida ms llamada la curva de retrasos o tiempos de respuesta, que representa el tiempo que requiere el servidor para completar una solicitud. Es de esperar que los tiempos de respuesta aumenten entre ms usuarios deba procesar el sistema, porque hay una competencia de recursos computacionales.

Encontrar las cuatro curvas mencionadas hasta ahora es, en esencia, el objetivo de las pruebas de estrs. Conociendo cul es el punto de quiebre del sistema se puede comparar esta cantidad con el nmero de usuarios por segundo que se espera atender en el ambiente de produccin, e incrementar los recursos del sistema si es necesario.

Cabe mencionar que existe otra forma de estresar el sistema que consiste en quitar temporalmente del servidor recursos de memoria, capacidad de procesamiento o ancho de banda. Por ejemplo, cuando ser quiere saber cmo reaccionar un balanceador de carga cuando uno de sus servidores se reinicie en media tarea de procesamiento [8].

2 Probando un sitio web¶

Teniendo una idea de qu es una prueba de estrs, podemos estudiar el comportamiento de un sistema real a modo de ejemplo. Para ello escogimos una pgina web llamada bloodpressurepulse.herokuapp.com dedicada a la visualizacin de datos. El sitio web toma archivos CSV almacenados en la nube y genera grficos interactivos usando Python Dash. Un vistazo de la interfaz principal se observa en la figura 2.

Figura 2. Pantallazo del sitio web.

Como explicamos en la seccin anterior, hay varias formas de estresar una pgina web, y en este ejemplo elegimos incrementar el nmero de usuario por segundo en lugar de introducir archivos CSV cada vez ms grandes.

Empezamos por definir los parmetros del tiempo de rampa (ramp-up period) y el total de usuarios que necesita la herramienta JMeter:

  • Tiempo de rampa (TR): 60 segundos.
  • Total de usuarios: 30 (i.e. el acumulado de usuarios o $\sum_{n=0}^{\text{TR}}{x[n]}$).

El tiempo de rampa es equivalente a nuestro parmetro $N$ de la seccin 1, y esta medida, junto con el total de usuarios define la pendiente de la curva de usuarios por segundo. Ambos parmetros se introducen a la herramienta JMeter para iniciar la prueba. Esta aplicacin fue escogida por ser gratuita y de amplio uso en la industria, aunque hay otras aplicaciones pagas, como Blaze Meter o Load Ninja, con mejores interfaces grficas. El cdigo de la solucin puede encontrarse en este enlace y las figuras 3-5 muestran los resultados obtenidos.

3 ¿Son los servidores web sistemas LTI?¶

Habiendo visto la teora bsica de las pruebas de estrs y habiendo ejecutado un ejemplo en un sitio real es momento de ver si podemos refinar nuestro modelo matemtico. Para ello nos preguntamos si es posible predecir el comportamiento del sistema procesando otro tipo de entradas, por ejemplo, un trapezio como el que muestra la figura 1 para las pruebas de carga. Para ello, acudimos a un tipo de sistemas llamados sistemas lineales e invariantes en el tiempo (LTI, por sus siglas en Ingls) que permiten caracterizar completamente el comportamiento de un sistema midiendo su respuesta al impulso. Para que un sistema sea LTI es necesario determinar si un cumple con las siguientes dos condiciones [3]:

  1. Cumplir con el principio de superposicin: $T(\alpha{x_1}+\beta{x_2}) = \alpha{T(x_1)}+\beta{T(x_2)}$ (escalamiento y aditividad).
  2. Ser invariante en el tiempo.

Empezamos tomando la ecuacin 2 y calculando el resultado del operador $T$ ($T=y[n]$) del lado izquierdo y derecho de la condicin 1. A modo de simplicacin hacemos $\alpha=\beta=1$ y $x_1[n]=x_2[n]=q$, como sigue:

$$ T(x_1[n]+x_2[n]) = \min(x_1[n]+x_2[n], q) = \min(2q,q) = q, $$$$ T(x_1[n])+T(x_2[n]) = \min(x_1[n], q) + \min(x_2[n], q) = \min(q,q) + \min(q, q) = 2q. $$

Observamos de inmediato que la ecuacin 2 no cumple la propiedad de aditividad porque $T(x_1[n]+x_2[n]) \neq T(x_1[n])+T(x_2[n])$, por lo tanto el sistema no es LTI, y no hace falta demostrar la condicin de invarianza en el tiempo. La condicin 1 se puede interpretar como que la salida tiene que ser proporcional a la entrada, por lo tanto, una entrada pequea da como resultado una salida pequea y viceversa, pero nuestro sentido comn y la experiencia nos dice que el sistema no da ms despus de haber alcanzado su capacidad mxima.

Es importante destacar que el tipo de modelo afecta la calidad de las estimaciones porque tambin pudimos haber usado $y[n] = x[n] - x[n-n_0]$, en lugar de la ecuacin 2, y con esto haber obtenido un sistema LTI (se deja al lector el ejercicio de demostrarlo), pero este modelo no sera acorde a la realidad como observamos a continuacion:

def respuestaRampaInocente(x, n0):
    N = len(x)
    y = np.zeros(N)
    for n in range(1,N):
        y[n] = x[n] - (x[n-n0] if 0<=n-n0 else 0)
    return y

y = respuestaRampaInocente(2*x,n0)
pylab.plot(2*x,'--',label='V/s')

pylab.plot(2*x-y,label='Errores por segundo')

Aca el haber procesado la entrada $x[n] = 2 \frac{q}{n_0} n u(n)$ habra duplicado la capacidad del sistema, lo cual es falso con recursos computacionales limitados. Como no es posible modelar un servidor web mediante un sistema LTI es necesario acudir a otras tcnicas de modelado no lineales [6,7].

Conclusiones¶

Las pruebas de estrs consisten en caracterizar el comportamiento del sistema ante un nmero creciente de usuarios virtuales por segundo (en forma de rampa) o una cantidad mayor de datos, utilizando recursos finitos o menos recursos computacionales que de costumbre. Este tipo de pruebas permite encontrar cuatro curvas llamadas: ancho de banda, errores por segundo, tasa de errores, y tiempos de respuesta. La ecuacin $y[n] = \min(x[n], q)$ es un modelo vlido que permite representar el comportamiento del sistema antes y despus del llegar al punto de quiebre. Esta ecuacin permiti determinar adems que los servidores web no son sistemas lineales e invariantes en el tiempo porque tienen una capacidad limitada de procesamiento y su salida no vara proporcionalmente a la entrada. Es necesario estudiar tcnicas de modelado no lineal para simular la salidad del sistema ante entradas usadas en otro tipo de pruebas, por ejemplo, entradas de forma trapezoidal empleadas en las pruebas de carga.
