Relacin de los patrones SOLID y los principios de programacin orientada a objetos

Juan M. Fonseca-Sols March/April 2025 10 min read

Introduccin

En la metodologa de trabajo SCRUM, el equipo de desarrollo de software incorpora funcionalidad nueva al sistema de forma incremental (cada dos semanas, por ejemplo) que posteriormente el equipo de pruebas y el cliente evalua para determinar si los requerimientos fueron satisfechos correctamente. Durante este proceso, los cambios realizados pueden afectar la funcionalidad pre-existente, por lo que es necesario correr pruebas de regresin, es decir, repetir las pruebas realizadas en las iteraciones pasadas. Como la ejecucin de estas pruebas resulta costozo, se recurre a la automatizacin; sin embargo, puede ocurrir tambin que los scripts de automatizacin pasados dejen de funcionar, entre otras razones, por un mal diseo de los mismos. Para evitar esto se utilizan cuatro patrones de diseo, llamados "SOLID", que permiten que lograr que la solucin de pruebas sea mantenible y escalable en el tiempo, procurando una "alta cohesin y bajo acoplamiento" entre sus componentes (GRASP) [1, 2]. El acrnimo SOLID se define como sigue:

  • S - Responsabilidad nica (single responsibility).
  • O - Abierto-cerrado (Open-closed).
  • L - Sustitucin de Liskov (Liskov substitution).
  • I - Segregacin de la interfaz (Interface segregation).
  • D - Inversin de la dependencia (Dependency inversion).

Estos cuatro patrones de diseo ya son ampliamente explicados en la literatura y en este ipython notebook no pretendemos hacer lo mismo, sino que ms bien nos enfocamos en describir su relacin con los cuatro principios bsicos de programacin orientada a objetos (OOP, por sus siglas en ingls): polimorfismo, abstraccin, encapsulamiento y herencia; pues ambos conceptos suelen confundirse.

Responsabilidad nica (encapsulamiento)

En este diseo cada componente de software (clase o mtodo) desempea o "encapsula" una sola tarea, lo que permite reemplazarlo sin tener que modificar otros archivos de cdigo. Este diseo tambin est relacionado con el diseo de segregacin de la interfaz.

from abc import ABC, abstractmethod

class Funcion(ABC):
    
    def __init__(self):
        pass
    
    @abstractmethod
    def evaluar(self, x:float):
        pass
    
    def evaluarArreglo(self, arreglo:list):
        return [self.evaluar(x) for x in arreglo]

Abierto-cerrado (polimorfismo)

En el diseo abierto-cerrado los componente de software estn abiertos para extender su funcionalidad pero cerrados para su modificarla; esto se cumple en el principio de polimorfismo, donde la funcionalidad de una clase padre (por ejemplo, TestBase o PageBase) se escribe una sola vez y son las clases hijas las encargadas de agregar funcionalidad, por ejemplo, atributos y mtodos nuevos o la implementacin de mtodos abstractos.

class Lineal(Funcion):
    
    def __init__(self, m:float, b:float):
        super()
        self.m = m
        self.b = b
    
    def evaluar(self, x:float):
        return self.m*x+self.b
    
class Cuadratica(Funcion):
    
    def __init__(self, a:float, b:float, c:float):
        super()
        self.a = a
        self.b = b
        self.c = c
    
    def evaluar(self, x:float):
        return self.a*x**2+self.b*x+self.c

Sustitucin de Liskov (herencia)

En este diseo los atributos y mtodos de la clase padre tambin estn presentes en sus clases hijas, y pueden ser invocados sin necesidad de realizar conversin explcita de datos; lo que se relaciona al principio de herencia.

Segregacin de la interfaz (encapsulamiento)

En el diseo de segregacin de la interfaz los componentes implementan roles documentados o "encapsulados" en una interfaz (por ejemplo, ejecutar operaciones de lectura, escritura y borrado en un repositorio). Es mejor tener "muchas interfaces de cliente especficas que una interfaz de propsito general).

import matplotlib.pylab as plt
import numpy as np

class Graficadora(ABC):
    
    def __init__(self):
        pass
    
    def graficar(self, arregloX:list, arregloY:list):
        plt.plot(arregloX, arregloY, label='Funcin Seno')
        plt.xlabel('Tiempo discreto (muestras)')
        plt.ylabel('Amplitud')
        plt.legend()

# en Python no existen las interfaces, pero se puede emular usando herencia mltiple
class Seno(Funcion, Graficadora):
    
    def __init__(self, Fs:float, F0:float):
        super()
        self.Fs = Fs
        self.F0 = F0
        
    def evaluar(self, x:float):
        return np.sin(2*np.pi*self.F0/self.Fs*x)
    
funcionSeno = Seno(8000, 440)
x = np.linspace(0,100,100)
funcionSeno.graficar(x, funcionSeno.evaluarArreglo(x))

Conclusiones

Los patrones de diseo resumidos en el acrnimo SOLID se relacionan con los principios de programacin orientada a objetos. Responsabilidad-nica y segregacin de la interfaz se relacionan al principio de encapsulamiento, pues las funciones se encapsulan en clases e interfaces. El diseo abierto-cerrado se relaciona con el principio de polimorfismo, pues son las clases hijas las que extienden la funcionalidad de las clases padres. El diseo de sustitucin de Liskov est ligado al de herencia, pues la clase hija hereda de la clase padre y puede usar sus mismos atributos y mtodos. Finalmente, el diseo de inversin de la dependencia se relaciona al principio de abstraccin, pues primero se piensa en la funcin que debe desempear un componente subordinado y luego en la implementacin para lograrlo.
