Relacin de los patrones SOLID y los principios de programacin orientada a objetos

Juan M. Fonseca-Sols March/April 2025 10 min read

Introduccin

En metodologas de trabajo incrementales, el equipo de desarrollo de software incorpora funcionalidad nueva al sistema de forma acumulativa (cada dos semanas, por ejemplo) que posteriormente el equipo de pruebas y el cliente evaluan para determinar si los requerimientos fueron satisfechos correctamente. Durante este proceso, los cambios realizados pueden afectar la funcionalidad pre-existente, por lo que es necesario correr pruebas de regresin, es decir, repetir las pruebas creadas en iteraciones pasadas. Como la ejecucin de estas pruebas resulta costozo, se recurre a la automatizacin; sin embargo, puede ocurrir tambin que los scripts de automatizacin dejen de funcionar; entre otras razones, por un mal diseo arquitectnico de los mismos. Para evitar esto se utilizan cuatro principios de diseo resumidos en el acrnimo "SOLID" que, favoreciendo una alta cohesin y un bajo acoplamiento entre los componentes (GRASP), facilitan el mantenimiento y escalabilidad de la solucin de pruebas [1, 2]:

  • S - Responsabilidad nica (single responsibility).
  • O - Abierto-cerrado (Open-closed).
  • L - Sustitucin de Liskov (Liskov substitution).
  • I - Segregacin de la interfaz (Interface segregation).
  • D - Inversin de la dependencia (Dependency inversion).

Estos cuatro principios de diseo ya son ampliamente explicados en la literatura y en este ipython notebook no pretendemos hacer lo mismo, sino que ms bien nos enfocamos en describir su relacin con los cuatro principios bsicos de programacin orientada a objetos (OOP, por sus siglas en ingls): polimorfismo, abstraccin, encapsulamiento y herencia; pues ambos conceptos suelen confundirse.

Responsabilidad nica (encapsulamiento)

En este principio, cada componente de software (clase o mtodo) desempea o "encapsula" una sola tarea, lo que permite reemplazarlo sin tener que modificar otros archivos de cdigo. Este principio tambin se relaciona con el OOP de segregacin de la interfaz, que explicamos ms adelante. Abajo, por ejemplo, definimos una clase "Funcion" cuya nica tarea es evaluar elementos del dominio y, por aparte, una clase "Graficador" encargado de dibujar funciones:

In[54]:
from abc import ABC, abstractmethod

class Funcion(ABC):
    
    def __init__(self):
        pass
    
    @abstractmethod
    def evaluar(self, x:float) -> float:
        pass
    
    def evaluarArreglo(self, xs:list) -> list:
        return [self.evaluar(x) for x in xs]

class Graficador(ABC):
    
    def __init__(self, leyenda:str, etiquetaX:str, etiquetaY:str):
        self.leyenda = leyenda
        self.etiquetaX = etiquetaX
        self.etiquetaY = etiquetaY
        pass
    
    def graficar(self, arregloX:list, arregloY:list, ) -> None:
        plt.plot(arregloX, arregloY, label=self.leyenda)
        plt.xlabel(self.etiquetaX)
        plt.ylabel(self.etiquetaY)
        plt.legend()

Abierto-cerrado (polimorfismo)

En el principio abierto-cerrado los componente de software estn abiertos para extender su funcionalidad pero cerrados para su modificarla; esto se cumple en el principio OOP de polimorfismo, donde la funcionalidad de las clases padre (por ejemplo, TestBase o PageBase) se escribe una sola vez y son las clases hijas las encargadas de agregar funcionalidad nueva, como atributos y mtodos adicionales o la implementacin de mtodos abstractos. A continuacin, por ejemplo, definimos dos clases hijas "FuncionLineal" y "FuncionCuadratica" que implementan la funcionalidad de la clase padre "Funcion":

In[55]:
class FuncionLineal(Funcion):
    
    def __init__(self, m:float, b:float):
        super()
        self.m = m
        self.b = b
    
    def evaluar(self, x:float) -> float:
        return self.m*x+self.b
    
class FuncionCuadratica(Funcion):
    
    def __init__(self, a:float, b:float, c:float):
        super()
        self.a = a
        self.b = b
        self.c = c
    
    def evaluar(self, x:float) -> float:
        return self.a*x**2+self.b*x+self.c

Sustitucin de Liskov (herencia)

En este principio los atributos y mtodos de la clase padre tambin estn presentes en sus clases hijas, y pueden ser invocados sin necesidad de realizar conversin explcita de datos (casting, en ingls); lo que se relaciona al principio OOP de herencia. Abajo, por ejemplo, creamos dos instancias de las clases hijas de "Funcion" e invocamos el mtodo "evaluarArreglo", definido solo en la clase padre.

In[56]:
funcionLineal = FuncionLineal(1, 0)
funcionCuadratica = FuncionCuadratica(1, 2, 3)
x = [0, 0.5, 1.0, 2.0, 4.0]

print(funcionLineal.evaluarArreglo(x))
print(funcionCuadratica.evaluarArreglo(x))

Segregacin de la interfaz (encapsulamiento)

En el principio de segregacin de la interfaz los componentes implementan roles documentados o "encapsulados" en una interfaz (por ejemplo, ejecutar operaciones de lectura, escritura y borrado en un repositorio). Es mejor tener "muchas interfaces de cliente especficas que una interfaz de propsito general". A continuacin, por ejemplo, definimos una nueva clase hija llamada "FuncionSeno" con la cual, abusando del principio de responsabilidad nica para efectos didcticos, podemos graficar en pantalla:

In[57]:
import matplotlib.pylab as plt
import numpy as np

# en Python no existen las interfaces, pero se pueden emular usando herencia mltiple
class FuncionSeno(Funcion, Graficador):
    
    def __init__(self, Fs:float, F0:float):
        super()
        self.Fs = Fs
        self.F0 = F0
        self.leyenda = 'Funcin Seno'
        self.etiquetaX = 'Tiempo discreto (muestras)'
        self.etiquetaY = 'Amplitud'
        
    def evaluar(self, x:float) -> float:
        return np.sin(2*np.pi*self.F0/self.Fs*x)
    
funcionSeno = FuncionSeno(Fs=8000, F0=440)
x = np.linspace(0,100,100)
funcionSeno.graficar(x, funcionSeno.evaluarArreglo(x))

Inversin de la dependencia (abstraccin)

Este principio propone que "se debe depender de abstracciones, no de implementaciones", es decir, que primero se disea la firma del mtodo (parmetros, valores de retorno y funcin) en el componente subordinado y luego se piensa en su implementacin. Conclusiones

Los principios de diseo resumidos en el acrnimo SOLID se relacionan con los principios de programacin orientada a objetos OOP. Responsabilidad-nica y segregacin de la interfaz se relacionan al principio de encapsulamiento, pues las funciones se encapsulan en clases e interfaces. El principio abierto-cerrado se relaciona con el de polimorfismo, pues son las clases hijas las que extienden la funcionalidad de las clases padres. El principio de sustitucin de Liskov est ligado al de herencia, pues la clase hija hereda de la clase padre y puede usar los mismos atributos y mtodos. Finalmente, el principio de inversin de la dependencia se relaciona al principio de abstraccin, pues primero se piensa en la funcin que debe desempear un componente subordinado y luego en la implementacin para lograrlo.
